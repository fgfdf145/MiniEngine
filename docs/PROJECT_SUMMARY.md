# MiniEngine 阶段性总结

## 1. 项目当前定位

MiniEngine 目前已经从最初的图形渲染验证，发展为一个以编辑器工作流为中心的 3D 场景编辑原型。

从现有代码看，它的核心定位更接近：

- 一个基于 SDL3 + Vulkan + ImGui 的轻量级编辑器框架
- 一个支持 glTF 2.0 导入、场景摆放、材质侧车编辑的模型场景工具
- 一个带有引擎雏形的数据层、渲染层和平台层的 MiniEngine 原型

它现在还不是一个完整意义上的游戏引擎。更准确地说，当前阶段已经完成了“编辑器闭环”，但还没有完成“运行时引擎闭环”。

## 2. 当前整体架构

项目采用自上而下的分层结构：

1. `app`
2. `engine/application`
3. `engine/renderer`
4. `engine/logic`
5. `engine/scene`
6. `engine/platform`
7. `engine/core`

其中真正承担主流程的是 `application -> renderer -> logic/scene -> platform/core` 这条链路。

### 2.1 启动链路

程序入口在 `app/main.cpp`。

启动流程如下：

1. 初始化日志系统
2. 打印 ImGuizmo / yaml-cpp / EnTT 的链接状态
3. 解析命令行参数
4. 创建 `EditorApplication`
5. 创建窗口与渲染后端
6. 进入主循环
7. 分发 SDL 事件并持续绘制帧

当前支持的启动参数：

- `--model <path>`：启动后直接加载指定模型
- `--frames <count>`：限制运行帧数，适合调试或自动化场景

### 2.2 模块职责概览

#### `app`

负责程序入口，仅保留最薄的一层启动代码。

#### `engine/application`

负责应用生命周期和主循环控制。

当前 `EditorApplication` 的职责包括：

- 解析启动参数
- 创建共享渲染状态
- 创建窗口
- 创建渲染后端
- 驱动整个应用主循环

这一层比较干净，定位明确，是一个标准的应用壳。

#### `engine/core`

提供基础能力，目前主要包括：

- 日志系统
- 输入系统

其中输入系统已经不只是简单键盘封装，而是统一管理：

- 键盘按下、抬起、重复
- 鼠标视角控制和平移
- 手柄连接、断开、按钮、摇杆、扳机
- 输入日志缓存
- 视口交互区域约束

这部分已经具备“编辑器级输入系统”的雏形。

#### `engine/platform`

提供平台相关的薄封装，目前主要包括：

- SDL 窗口创建与销毁
- 文件对话框

窗口层负责创建 Vulkan 可用窗口。
文件对话框层负责模型、贴图、场景文件的选择与保存。

#### `engine/scene`

定义场景世界的数据接口和基础组件，是数据层而不是行为层。

目前主要内容包括：

- `TagComponent`
- `TransformComponent`
- `ModelComponent`
- `ModelImportedMaterialInfo`
- `ModelImportedSubmeshInfo`
- `MaterialTextureBlendGraph`
- `ISceneWorld`
- 世界单位常量

这层很重要，因为它把“编辑器显示的数据”和“渲染真正依赖的数据”分离成了稳定结构。

#### `engine/logic`

当前的核心是 `EditorScene`，它既是编辑器逻辑层，也是场景世界接口实现。

它承担了以下职责：

- 基于 EnTT 管理实体
- 维护实体顺序
- 维护当前选择实体
- 创建和删除实体
- 管理 Transform / Model / Tag 数据
- 从 YAML 加载场景
- 保存场景到 YAML
- 输出场景 YAML 预览
- 维护 gizmo 参数
- 维护场景默认配置

换句话说，`EditorScene` 目前已经是编辑器里的主场景模型。

#### `engine/renderer`

这是当前项目最重、最核心的一层。

它不只是“渲染器”，实际上同时承担了：

- 编辑器后端流程
- 资产导入流程
- 场景到可绘制数据的转换
- 模型加载
- 材质 sidecar 维护
- ImGui 编辑器界面
- 渲染后端抽象
- Vulkan 实现

这一层内部又可以继续分成几块：

1. 编辑器后端控制
2. 编辑器 UI
3. 模型与材质处理
4. 渲染后端抽象
5. Vulkan 实现

##### 编辑器后端控制

`EditorRenderBackendBase` 是当前真正的工作流中枢。

它负责：

- 初始化编辑器共享状态
- 加载编辑器设置
- 初始化默认场景
- 逐帧驱动输入和相机
- 处理 UI 发出的操作请求
- 执行模型导入、场景加载、场景保存
- 执行实体创建、删除
- 执行材质 sidecar 更新
- 执行视口拖拽放置模型
- 把 `EditorScene` 转成 CPU 渲染子网格

可以把它理解为“编辑器应用服务层 + 渲染准备层”。

##### 编辑器 UI

`EditorUiController` 已经形成一套完整编辑器界面。

当前主要窗口包括：

- `Viewport`
- `Scene`
- `Asset Manager`
- `Camera`
- `Theme`
- `Input Monitor`
- `Model Preview`
- `Material Preview`

其功能已经不是简单面板，而是形成了完整交互闭环：

- 停靠布局和工具栏
- 视口交互
- 资产浏览
- 资产右键菜单
- 资产复制、粘贴、重命名、删除
- 模型导入
- 模型预览
- 材质预览和编辑
- 场景加载和保存
- 主题编辑
- 输入监视

##### 模型与材质处理

这一块已经很有“资源管线”的味道。

当前实现包括：

- 仅支持 `glTF 2.0` 模型（`*.gltf` / `*.glb`）
- 基于 `tinygltf` 提取网格、材质、贴图路径
- 自动进行单位换算到米
- 自动三角化与子网格收集
- 自动补法线
- 自动补切线
- 生成导入后的资产清单 `.miniengine_asset.yaml`
- 为每个材质生成 `.material.yaml`
- 从 sidecar 覆盖材质贴图与简化纹理图设置

项目里已经出现两层资源描述：

1. 模型源文件本身的数据
2. MiniEngine 自己的 sidecar 元数据

这说明资产系统已经进入“可持续扩展”的阶段，而不是一次性加载。

##### 渲染后端抽象

`engine/renderer/rhi` 提供了后端抽象接口，但当前仍处于轻量阶段。

现在的特点是：

- 抽象存在
- 工厂存在
- 枚举类型存在
- 但只有 Vulkan 一个真实实现

因此它更像“未来扩展点”，还不是完全成熟的 RHI 层。

##### Vulkan 实现

Vulkan 后端已经完成基础渲染闭环，当前能力包括：

- Vulkan 实例与设备创建
- 交换链管理
- Render Pass
- Uniform Buffer
- 命令录制与提交
- 纹理上传
- 图形管线
- 视口场景图层
- ImGui Vulkan 集成
- 场景先渲染到视口纹理，再嵌入编辑器 UI

纹理处理方面已经包含：

- 默认白色底色纹理
- 默认法线纹理
- 默认 metallic / roughness / occlusion / emissive / blend mask 兜底
- 贴图缓存
- 材质槽绑定

这意味着项目已经从“能画东西”进化为“能稳定承接编辑器工作流”。

## 3. 当前已经实现的功能闭环

### 3.1 编辑器启动与基础运行闭环

已经完成：

- 项目可构建
- 程序可启动
- SDL 窗口可创建
- Vulkan 渲染后端可初始化
- ImGui 可运行
- 主循环可稳定执行

这是整个项目的地基，已经比较稳。

### 3.2 场景编辑闭环

当前场景系统已经支持：

- 默认创建测试场景
- 管理多个实体
- 选中实体
- 新增实体
- 删除实体
- 编辑实体名称
- 编辑位移、旋转、缩放
- 使用 gizmo 在视口里直接操作
- 显示选中实体的 YAML 预览

这套功能已经足够支撑一个基础场景编辑器原型。

### 3.3 视口交互闭环

当前视口已支持：

- 显示渲染结果
- 鼠标点击选择实体
- 显示选择框与包围盒投影
- ImGuizmo 操作
- `W / E / R` 切换变换模式
- `F` 聚焦选中对象
- 视图操纵器
- 右键视角旋转
- 中键平移
- WASD 移动相机
- 手柄控制相机
- 拖拽模型到视口中直接放置

这说明视口已经从纯展示，变成了真正的编辑交互区域。

### 3.4 资产管理闭环

Asset Manager 已经具备：

- 浏览 `assets` 目录
- 识别模型、材质、场景文件
- 打开模型预览
- 打开材质预览
- 导入 glTF 模型
- 检测重复导入
- 复制资产
- 粘贴资产
- 重命名资产
- 删除资产

这部分的完成度比普通原型要高，说明项目已经明显朝编辑器产品形态演进。

### 3.5 模型导入闭环

模型导入流程目前已经打通：

1. 选择源 glTF 资源（`.gltf` 或 `.glb`）
2. 复制到 `assets/models/<bundle>/`
3. 生成 MiniEngine 资产清单
4. 解析材质与贴图信息
5. 复制材质贴图到本地资产目录
6. 为每个材质生成独立材质描述文件
7. 将导入结果提供给场景和渲染器使用

导入后的模型会在 `assets/models/` 下生成自己的 bundle、材质 sidecar 和贴图副本。

### 3.6 材质侧车编辑闭环

当前材质系统已经支持对导入模型的材质进行二次描述和回写，主要包括：

- 主层 PBR 贴图路径
- 法线 / 金属度 / 粗糙度 / AO / 自发光
- secondary layer
- blend mask
- blend factor

修改后会：

- 写回模型 sidecar
- 写回材质 sidecar
- 刷新引用该模型的场景文件
- 重新构建渲染数据

虽然还不是真正节点式材质编辑器，但已经具备“可编程材质资源定义”的早期形态。

### 3.7 场景序列化闭环

当前场景已经支持 YAML 的读取、应用和保存。

可序列化内容包括：

- 实体列表
- 实体 tag
- 模型显示名
- 模型源路径
- base color override
- transform
- gizmo 设置
- 当前选中实体索引

这使得编辑器状态不再是内存态，而是可落盘、可回放、可迭代。

### 3.8 编辑器设置持久化闭环

当前已经支持将编辑器设置存入 `miniengine.settings.json`。

内容包括：

- UI 缩放
- 各窗口显隐状态
- Theme 颜色

这一步虽然看起来不大，但很重要，因为它意味着“工具体验”已经进入考虑范围。

## 4. 当前代码结构的优点

### 4.1 分层总体是清晰的

虽然 `renderer` 里职责偏重，但顶层模块边界还是可识别的：

- `core` 做基础设施
- `platform` 做平台封装
- `scene` 做数据定义
- `logic` 做场景逻辑
- `renderer` 做 UI、资源转换和渲染
- `application` 做启动和生命周期

这种结构为后续继续拆分提供了基础。

### 4.2 数据结构已经比较稳定

`ModelComponent`、`TransformComponent`、`GizmoSettings`、`SerializedSceneData`、`ModelImportedMaterialInfo` 这些结构已经能稳定串起：

- UI
- 场景
- 资源
- 渲染
- 序列化

这说明当前项目的“核心数据模型”已经逐渐成型。

### 4.3 编辑器工作流是闭环的

目前项目最成熟的不是底层引擎，而是编辑器工作流。

从用户角度已经能完成：

- 打开程序
- 导入模型
- 在场景中放置模型
- 调整相机和变换
- 编辑材质
- 保存场景

这是非常有价值的阶段成果。

### 4.4 资源 sidecar 设计方向是正确的

导入模型后不直接修改源 glTF，而是生成自己的 sidecar 和材质文件，这个方向是对的。

它为未来扩展以下能力打下了基础：

- 导入设置
- 资源重导入
- 材质实例
- 资源版本迁移
- 批处理

## 5. 当前结构中的主要问题与边界

### 5.1 `renderer` 模块职责过重

这是当前最明显的问题。

`engine/renderer` 现在混合了：

- 编辑器应用逻辑
- 资产导入逻辑
- 资源元数据维护
- 场景到渲染数据转换
- ImGui UI
- Vulkan 渲染后端

这会带来几个问题：

- 文件过大，尤其 `editor_ui.cpp` 和 `editor_backend_base.cpp`
- 修改一处逻辑容易牵动多层
- 不利于未来加第二渲染后端
- 不利于未来拆出 runtime 模式

### 5.2 RHI 还没有真正独立

虽然有 `rhi/backend.h` 和 `rhi/factory.cpp`，但目前实际还是 Vulkan 直连式实现。

如果未来要支持：

- DirectX 12
- Metal
- OpenGL 兼容层
- Headless/offscreen 渲染

当前抽象还不够。

### 5.3 `EditorScene` 仍是编辑器专用场景

`EditorScene` 很实用，但它仍偏向编辑器数据模型，而不是通用运行时世界。

例如：

- 没有系统调度层
- 没有组件生命周期规则
- 没有游戏对象运行逻辑
- 没有 runtime-only 与 editor-only 数据分层

这意味着它很适合作为编辑器场景，但还不能直接承担完整运行时世界。

### 5.4 资产系统还处于“原型资源管线”

虽然 sidecar 已经出现，但目前仍缺少：

- 统一 `AssetDatabase`
- 资源 UUID
- 资源引用追踪
- 资源导入设置对象
- 重导入策略
- 依赖图

现在更像“文件驱动型资源系统”，而不是“数据库驱动型资源系统”。

### 5.5 引擎运行时能力尚未展开

当前还缺少典型引擎模块：

- 游戏运行时模式
- 脚本系统
- 动画系统
- 物理系统
- 音频系统
- 资源热更新
- 任务系统
- 场景实例化与运行态分离

因此从“编辑器原型”迈向“引擎产品”，还有明显距离。

## 6. 当前阶段最准确的项目画像

结合代码现状，可以把项目描述为：

> 一个基于 SDL3、Vulkan、Metal、ImGui、EnTT、yaml-cpp 与 tinygltf 构建的 3D 编辑器原型。
> 它已经完成模型导入、场景编辑、材质 sidecar、视口交互和 Vulkan 渲染闭环，
> 当前重点在编辑器工作流和资源组织，而不是完整游戏运行时。

如果要更口语一些，可以概括为：

- 已经是“能用的编辑器原型”
- 还不是“完整的游戏引擎”
- 最成熟的是编辑器和资源导入链路
- 最需要继续演进的是运行时能力和模块拆分

## 7. 推荐的下一阶段整理方向

### 7.1 第一优先级：先拆职责，再扩功能

当前最建议做的不是继续堆功能，而是先把已有逻辑按职责拆开。

优先拆分方向建议：

- 从 `engine/renderer` 中拆出 `editor` 模块
- 从 `engine/renderer` 中拆出 `asset` 或 `asset_pipeline` 模块
- 将 `editor_backend_base.cpp` 中的业务操作整理为独立 service
- 将 `editor_ui.cpp` 拆成多窗口文件

目标是让结构从“一个大模块负责所有事”变成“编辑器、资产、渲染协作”。

### 7.2 第二优先级：补全资源系统骨架

建议逐步引入：

- 资源 ID
- 资源注册表
- 导入记录
- 引用关系
- 重导入入口

这样后续材质、纹理、模型、场景才能形成统一资产体系。

### 7.3 第三优先级：建立运行时与编辑器的边界

建议明确区分：

- Editor-only 数据
- Runtime 数据
- 序列化格式
- 运行态实例

这一步很关键，因为它决定项目未来是“编辑器工具”还是“编辑器 + 引擎”。

### 7.4 第四优先级：完善渲染抽象

如果未来目标是长期演进，建议逐步补齐真正的 RHI 边界：

- buffer / texture / pipeline / command 的抽象接口
- scene submission 抽象
- viewport/render target 抽象

否则未来 Vulkan 代码会继续侵入上层。

## 8. 当前阶段总结

截至目前，MiniEngine 已经完成以下关键跨越：

- 从单纯渲染验证，进入编辑器原型阶段
- 从只会加载数据，进入可编辑、可保存、可导入的工作流阶段
- 从依赖源模型，进入拥有自身 sidecar 元数据的资源组织阶段
- 从单帧绘制，进入完整的编辑器交互循环阶段

当前最值得肯定的，不是某一个单点技术，而是整条链路已经打通：

- 资产进入项目
- 场景引用资产
- 编辑器操作场景
- 渲染器消费场景
- 设置与场景可持久化

这说明项目已经具备非常扎实的下一阶段演进基础。

后续如果继续沿着“结构拆分 + 资源系统 + runtime 边界”三个方向推进，MiniEngine 会从“功能已经不少的原型”，逐步走向“真正可持续迭代的引擎项目”。
